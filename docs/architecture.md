# Kura architecture & monorepo plan

Goal: extract the framework base from `examples/docs` so example apps **consume
`@kurajs/*` packages** instead of hand-rolling nav/ToC/search/agent surface. An app
should be ~content + a few lines of wiring; everything else comes from the framework.

## Layering

Kura is a **layer on June** (like Starlight on Astro, Fumadocs on Next.js): June is the
web framework (routing, RSC, projections, MCP, deploy); Kura adds the *knowledgebase +
docs* conventions, components, and agent surface on top.

```
┌─────────────────────────────────────────────────────────────┐
│ App  (examples/docs)  — content/*.md + thin wiring + config   │
├─────────────────────────────────────────────────────────────┤
│ @kurajs/docs   the docs framework on June                     │
│   headless: nav tree · ToC · prev/next · KB build/load ·      │
│             doc/search route factories · agent action factories│
│   ui:       DocsShell · DocsPage · SearchResults · theme.css  │
├───────────────────────────────┬─────────────────────────────┤
│ @kurajs/core  retrieval engine │ @kurajs/transformers (embed) │
│   (zero-dep, portable)         │ @kurajs/workers-ai (future)  │
├───────────────────────────────┴─────────────────────────────┤
│ June  (@junejs/core, @junejs/server)  — the web framework     │
└─────────────────────────────────────────────────────────────┘
```

**Principles**
- `@kurajs/core` stays **zero-dependency, June-agnostic, Workers-safe** (the data heart).
- The June integration + UI lives **above** core, with `@junejs/core` + `react` as peer
  deps. Core never imports June or React.
- **Convention over configuration**: an app declares its content source + embedder once;
  routes are thin re-exports of framework handlers.
- **Agent surface is first-class**: every Kura site ships `.md`/`.json`/`llms.txt` (free
  from June) + `search_docs`/`get_page` MCP tools (from the framework). "humans and
  agents" is built in, not per-app.

## Packages

| package | role | deps | status |
|---|---|---|---|
| `@kurajs/core` | retrieval engine: `Kb`, `Embedder`, config types | none | ✅ exists |
| `@kurajs/transformers` | local bge-m3 embedder | `@huggingface/transformers` (peer) | ✅ exists |
| `@kurajs/workers-ai` | Cloudflare Workers AI embedder | — | ▢ future |
| **`@kurajs/docs`** | **the docs framework on June** (headless + ui) | `@kurajs/core`; peer `@junejs/core`, `react` | ▢ NEW |
| `@kurajs/cli` | `kura` CLI: local index/search, `kura index` (build search index) | core, transformers | ✅ exists (extend) |
| `create-kura` | scaffolder → a `@kurajs/docs` app | — | ✅ published 0.0.1 |

`@kurajs/docs` uses **subpath exports** to keep headless vs UI separable now and
splittable later (into `@kurajs/docs-core` + `@kurajs/docs-ui`, Fumadocs-style) if a
headless/custom-UI audience appears:

- `@kurajs/docs` — `defineDocs()`, route factories, nav/ToC/search helpers, action factories (no React).
- `@kurajs/docs/ui` — `DocsShell`, `DocsPage`, `SearchResults` + client scripts (React).
- `@kurajs/docs/theme.css` — the stylesheet.

## What moves out of `examples/docs` → `@kurajs/docs`

| current app file | becomes |
|---|---|
| `app/_nav.ts` (groups, prev/next, ToC, slugify, processHtml) | `@kurajs/docs` headless |
| `app/_kb.ts` (build/load search index from content) | `@kurajs/docs` `createSearch()` |
| `app/_view.tsx` (DocsShell, DocsPage, scripts) | `@kurajs/docs/ui` |
| `app/global.css` | `@kurajs/docs/theme.css` |
| `app/actions.ts` (search_docs, get_page) | `@kurajs/docs` action factories |
| `app/page.tsx`, `app/docs/[slug]/page.tsx`, `app/search/page.tsx` | thin re-exports of framework route factories |
| `scripts/build-index.mjs` | `@kurajs/cli` → `kura index` |

App keeps: `content/docs/*.md`, `june.config.ts`, generated `app/_content.ts`, a small
`app/_kura.ts` wiring file, thin route files, and `app/global.css` (just imports the theme).

## The integration seam (how a June app uses Kura)

The framework can't inject route files (June routing is file-based, `app/` scanned), so
each route file stays — but becomes a ~5-line re-export. The framework is **parameterized
by the app's content collection + embedder** (the generated `_content.ts` is app-specific).

```ts
// app/_kura.ts — wire the framework to THIS app, once
import { defineDocs } from "@kurajs/docs";
import { transformers } from "@kurajs/transformers";
import { DOCS, doc } from "./_content";           // generated by `june gen`

export const kura = defineDocs({
  content: { DOCS, doc },
  embedder: transformers(),                        // swap for workersAI() on CF
  sections: ["開始", "核心概念", "進階"],          // sidebar order
  site: { name: "Kura Docs", brand: "蔵 Kura" },
});
```

```ts
// app/docs/[slug]/page.tsx — thin
import { kura } from "../../_kura";
export const loader = kura.doc.loader;
export const md = kura.doc.md;
export const json = kura.doc.json;
export const metadata = kura.doc.metadata;
export default kura.doc.View;
```

```ts
// app/actions.ts — agent tools, free
import { kura } from "./_kura";
export const search_docs = kura.actions.search_docs;
export const get_page = kura.actions.get_page;
```

```css
/* app/global.css */
@import "@kurajs/docs/theme.css";
/* app-specific overrides below */
```

That's the whole app: content + `_kura.ts` + ~4 thin route files + a CSS import.

## CSS / theme delivery (decision)

June auto-links `app/global.css` and processes it with lightningcss. Preferred:
`@import "@kurajs/docs/theme.css"` (verify lightningcss resolves node_modules imports in
June). Fallback if not: `DocsShell` injects the theme via an inline `<style>` (no CSS-
pipeline dependency). **To verify during implementation.**

## The two examples after extraction

- **`examples/docs`** — gutted to the thin shape above; the showcase of `@kurajs/docs`.
- **`examples/search`** — stays a *minimal* "KB on June" example using `@kurajs/core` +
  `@kurajs/transformers` directly over **SQLite** (different shape: a search box over a DB,
  not a docs site). Demonstrates the engine without the docs UI. Largely unchanged.

## Migration plan (incremental, keep it green)

1. Create `@kurajs/docs` package skeleton (build, exports map, peer deps).
2. Move `_nav.ts` → headless; add a `defineDocs()` that returns nav/ToC helpers. Port
   `examples/docs` to import them; verify it still runs.
3. Move `_view.tsx` + `global.css` → `@kurajs/docs/ui` + `/theme.css`. Re-point the app.
4. Move `_kb.ts` search wiring → `createSearch()` (content source + embedder); move
   index-building into `kura index` (CLI). Re-point.
5. Move `actions.ts` → action factories. Re-point. Confirm `/mcp` still lists the tools.
6. Collapse the app routes to thin re-exports; delete the moved files. Re-verify end-to-end
   (browse, ToC, search, copy-md, MCP) in the browser.
7. Update `create-kura` to scaffold this shape.

## Decisions (resolved)

1. ✅ **One package** `@kurajs/docs` (subpath exports) now; split to `-core`/`-ui` later.
2. ✅ **Index building → `kura index` (extend existing `@kurajs/cli`)**, not a per-app
   script. No new package; workspace-linked `kura` bin in dev, publish later.
3. ✅ **Config = ONE declarative `kura.config.ts`; NO `app/_kura.ts`.** June supports
   **optional catch-all routes `[[...slug]]`** (verified `router.ts`), so the whole docs
   site is **one scaffolded route file** `app/docs/[[...slug]]/page.tsx` that imports
   `kura.config` + the generated `_content` + the framework handler and wires them — the
   binding lives there, so a separate `_kura.ts` isn't needed.
   - `kura.config.ts` (top-level, declarative only): `sections`, `theme`, `embedder`,
     agent toggles — the only file users edit.
   - Thin glue (`app/docs/[[...slug]]/page.tsx`, `app/actions.ts`) is **scaffolded by
     create-kura** — users never hand-write it.
   - Why glue can't be zero today: June discovers routes by scanning `app/`; a node_modules
     package can't inject files there (Next/Fumadocs-shaped). **North star: add a June
     route-mount / integration API** (what Astro has, enabling Starlight) → then
     `kura.config.ts` could register the catch-all + actions and the `app/` glue disappears
     entirely. June is ours, so this is a feasible future enhancement.
4. ✅ Embedder stays **injected** (local `transformers()` ↔ cloud `workersAI()`) for the
   local↔Workers parity.

## Config mental model (direction — refine later)

`kura.config.ts` is the **primary, user-facing config** — the mental model is "I'm using
Kura," not "I'm using June." June is the engine underneath. So `june.config.ts` should be
**optional / implied** by Kura (e.g. a Kura CLI `kura dev`/`kura build` wrapping June and
synthesizing June's config from `kura.config.ts`, or a scaffolded 1-line `june.config.ts`
shim, or — north star — the June integration API). Exact mechanism: discuss later; for now
keep a thin `june.config.ts` and treat `kura.config.ts` as the surface users edit.

## Resolved during implementation

- **Theme delivery = component-injected `<style>`.** `@kurajs/docs/ui` ships the theme as a
  string (`themeCss`) injected by `<DocsShell>` — no CSS-pipeline dependency, Workers-safe,
  guaranteed to load. The app's `app/global.css` is just for overrides (cascades after).
  (Chosen over `@import` to avoid depending on lightningcss node_modules resolution.)
